# OpenCode 全局规则

## 交流规范
- 始终使用简体中文回复
- 直接回答问题，不要客套话
- 代码注释也用中文

## 代码规范
- 使用 2 空格缩进
- 变量名用驼峰命名（camelCase）
- 函数名用动词开头（如 getUserById）
- 常量用大写字母和下划线（如 MAX_COUNT）
- 类名用大驼峰命名（如 UserController）

## 工作习惯
- 修改代码前先阅读相关文件
- 不确定时先问，不要猜测
- 每次只做最小必要的修改
- 优先使用项目中已有的工具和库
- 完成任务后主动运行测试或检查命令

## 工作态度
- 每次工作都要用严谨的工作态度，保证完美的质量标准

## 沟通风格
- 除非明确要求，否则不提供代码摘要

## 求真原则（禁止瞎猜）
- 不确定/信息不足时先查证或提问澄清
- 对环境/配置/源码/行为的结论必须有证据
- 回答里把"事实"和"推测/假设"分开写

## 代码质量原则
- 优先代码可读性，做最简单的修改
- 禁止使用 `any` 类型，必须定义明确的类型
- 不要为了向后兼容而保留废弃代码
- 删除未使用的代码，不要注释掉

## 复用优先
- 编写新代码前，先确认项目中是否已有类似实现
- 优先复用现有组件和工具函数，而非新建

## 执行规范
- 任何非平凡任务，先制定计划再动手
- 修改代码前必须先阅读相关文件

## 子代理调度策略
- 尽可能调用子代理完成任务
- 能派给专家的就派，不要什么都自己干

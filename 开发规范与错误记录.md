# 开发规范与错误记录

## 🚨 严重错误记录（必须避免）

### 错误1：重命名文件导致编辑器引用丢失
- **错误行为**：将 `GameConfig.js` 重命名为 `001_GameConfig.js`
- **严重后果**：
  - Cocos Creator 的 `.meta` 文件 UUID 关联断开
  - 所有节点绑定丢失
  - 脚本组件引用失效
  - 用户在编辑器中配置的内容全部丢失
- **根本原因**：不了解 Cocos Creator 依赖 `.meta` 文件的 UUID 关联
- **教训**：**绝对不能重命名已有文件**

### 错误2：批量修改多个文件
- **错误行为**：一次性修改多个脚本文件的 `require` 语句
- **后果**：代码混乱，难以追踪问题，容易遗漏
- **教训**：**每次只修改一个文件，确认无误后再继续**

### 错误3：删除调试代码时误删功能代码
- **错误行为**：清理日志时不小心删除了 `gameInfoManagerScript` 初始化代码
- **后果**：功能失效，需要重新添加
- **教训**：**删除前仔细检查，确认只删除调试代码**

### 错误4：添加重复代码
- **错误行为**：在文件中添加了重复的代码块
- **后果**：语法错误，编译失败
- **教训**：**添加代码前先检查是否已存在**

### 错误5：预制体结构理解错误（反复犯错）
- **错误行为**：用户要求 Chess 和 Shadow 同级，但我多次做成父子关系
- **后果**：
  - 用户反复纠正
  - 浪费时间
  - 破坏用户信任
- **根本原因**：
  - 没有理解"同级"和"父子"的区别
  - 没有确认现有代码的依赖关系
  - 急于修改，没有先问清楚
- **教训**：
  - **必须先问清楚再修改**
  - **必须理解现有代码依赖**
  - **不能假设用户的意思**

### 错误6：预制体循环嵌套
- **错误行为**：给 Shadow 节点也添加了 PrefabInfo
- **后果**：报"预制体不能循环嵌套"错误
- **根本原因**：
  - 不理解 PrefabInfo 的作用
  - 复制粘贴时没有检查
- **教训**：
  - **PrefabInfo 只能有一个，指向根节点**
  - **子节点不需要 PrefabInfo**

### 错误7：没有先问就直接修改
- **错误行为**：多次直接开始改文件，没有先确认方案
- **后果**：
  - 改错方向
  - 用户不满意
  - 需要反复修改
- **根本原因**：
  - 急于解决问题
  - 没有遵循"先问再做"原则
- **教训**：
  - **先读取当前状态**
  - **分析问题**
  - **提出方案**
  - **等用户确认后再修改**

## 📋 操作规范（必须遵守）

### 修改代码前
1. **说明要做什么**
2. **说明可能的影响**
3. **等用户确认后再执行**

### 文件操作
- ✅ 可以创建新文件
- ✅ 可以修改现有文件内容
- ❌ **绝对不能重命名已有文件**
- ❌ **绝对不能删除已有文件**

### 代码修改
- ❌ 不批量修改多个文件
- ❌ 不一次性添加大量代码
- ✅ 每次只修改一个文件
- ✅ 修改后立即检查语法

### 删除代码
- ❌ 不随意删除代码
- ✅ 删除前确认只删除调试代码
- ✅ 删除后检查是否影响功能

### 预制体修改（特别重要）
- ❌ **不能直接修改**，必须先确认方案
- ❌ **不能给子节点添加 PrefabInfo**
- ❌ **不能修改根节点的组件引用**
- ✅ 先读取当前结构
- ✅ 分析现有代码依赖
- ✅ 提出方案并等用户确认
- ✅ PrefabInfo 只能有一个，指向根节点

## 🔧 Cocos Creator 特殊注意

### .meta 文件
- Cocos Creator 使用 `.meta` 文件存储 UUID
- UUID 用于关联脚本和编辑器中的绑定
- **重命名或删除文件会导致 UUID 失效**
- **所有绑定都会丢失**

### 脚本加载
- 脚本文件名影响加载顺序
- 全局变量需要在使用前加载
- `window.XXX = XXX` 导出为全局变量

### 节点绑定
- 编辑器中的属性绑定依赖 `.meta` 文件
- 脚本组件挂载依赖文件名
- **修改文件名会导致所有绑定丢失**

### 错误8：粒子系统纹理创建和属性设置问题
- **错误行为**：程序化创建粒子系统时，多次尝试失败
- **遇到的问题**：
  1. `ps.play()` 方法不存在（Cocos 2.4.x 没有此方法）
  2. `cc.ParticleSystem.EMISSION_MODE.RADIAL` 属性不存在
  3. `ps.texture` 是只读属性，不能直接设置
  4. 使用 `ps.spriteFrame` 才是正确方式
  5. `Texture2D.initWithElement(canvas)` 创建的纹理不被识别
  6. 需要先将 Canvas 转换为 Image 对象，再创建 Texture2D
  7. `SpriteFrame.setTexture()` 返回 false 说明纹理设置失败
  8. 必须使用 Image.onload 回调确保纹理加载完成
  9. Battle.js 在 ParticleManager 加载完成前就调用了 `playAmbientParticles()`
  10. 粒子节点位置设置错误（需要考虑父节点坐标）
- **根本原因**：
  - 不了解 Cocos Creator 2.4.x 的粒子系统 API
  - 不了解程序化创建纹理的正确流程
  - 不了解异步加载的时序问题
  - 不了解坐标系统的嵌套关系
- **正确流程**：
  ```javascript
  // 1. 创建 Canvas 并绘制
  var canvas = document.createElement('canvas');
  var ctx = canvas.getContext('2d');
  // ...绘制内容...
  
  // 2. 转换为 Image 对象（异步）
  var img = new Image();
  img.onload = function() {
      // 3. 创建 Texture2D
      var texture = new cc.Texture2D();
      texture.initWithElement(img);
      
      // 4. 创建 SpriteFrame
      var spriteFrame = new cc.SpriteFrame();
      spriteFrame.setTexture(texture, cc.rect(0, 0, w, h), false, cc.v2(0, 0), cc.size(w, h));
      
      // 5. 设置粒子纹理（使用 spriteFrame，不是 texture）
      ps.spriteFrame = spriteFrame;
      
      // 6. 配置粒子属性并播放
      ps.resetSystem();
  };
  img.src = canvas.toDataURL('image/png');
  ```
- **教训**：
  - **粒子纹理必须使用 `ps.spriteFrame`，不是 `ps.texture`**
  - **Canvas 转纹理需要先转 Image 对象**
  - **必须在 Image.onload 回调中处理纹理**
  - **不要在组件初始化时就调用粒子播放，等待纹理加载完成**
  - **坐标系统要考虑父节点位置**
  - **参考官方 API 文档，不要凭猜测**
